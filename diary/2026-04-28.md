## What got built

### AlwaysFoldBot
The opposite of yesterday's `AlwaysCall`. Folds whenever it's facing a bet, checks when it isn't (folding for free is illegal in pokerkit). Trivial bot, 8 lines, but it pins one end of the spectrum: anyone who beats it does so by walking away with the blinds it forfeits every orbit.

### Stable UI seats (the runner-side seat translation)
Yesterday the bots rotated around the table because pokerkit's button is always at engine seat `n-1`, and the simplest way to rotate the button was to rotate which bot occupies each engine seat. That made the table hard to follow — the LLM bot was at seat 5 one hand and seat 0 the next.

The fix is a clean separation:
- **UI seat** = the bot's index in `self._bots` (fixed for the whole session — this is what the browser renders).
- **Engine seat** = what pokerkit sees (rotates each hand so the button moves).
- The runner is the boundary. Everything outbound on the websocket is in UI-seat space; observations passed into bots stay in engine-seat space (bots don't need to know the UI exists).

In code, the per-hand mapping is one short table:

```python
button_idx = self._hand_id % n_active
e2u = [active_ui[(button_idx + 1 + e) % n_active] for e in range(n_active)]
```

…plus two helpers `list_to_ui` / `dict_to_ui` for inflating engine-indexed lists (size `n_active`) and dicts (keyed by engine seat) to the UI's full size-`n_total` view, with sat-out seats filled in with safe defaults.

### Right-click sit-out / sit-in
Two pieces: the WS becomes bidirectional, and the runner gets a "who's playing this hand" abstraction.

**Wire protocol**: client sends `{"cmd": "sit_out" | "sit_in", "seat": N}`. The runner mutates a `_sitting_out: set[int]` and broadcasts a new `seats_update` event so any other open tab sees the change immediately.

**Engine resizing**: each `_play_one_hand` snapshots the active set at hand start. If fewer than 2 are active it clears an asyncio Event and waits. Otherwise it builds a `n_active`-player engine, rotates the button through `active_ui` (not through all 6 UI seats), and inflates engine-indexed payloads back up to `n_total = 6` for the broadcast so the frontend never has to bounds-check.

**UX**: a sit-out toggle takes effect at the *next* hand. The seat dims and shows a dashed-border "sitting out" badge immediately, even mid-hand — a small white lie (the bot still acts in the current hand) but it makes clicks feel responsive.

## Bugs / surprises

### The right-click-menu-flies-away bug

Most interesting bug today. Reported as: "sometimes when I right-click, the menu displays far away."

The menu was using `position: fixed` and setting `left: ${e.clientX}px; top: ${e.clientY}px`. By spec that's relative to the viewport. So why was it wrong sometimes?

Because `position: fixed` is **not** always relative to the viewport — it's relative to the nearest ancestor that has a `transform`, `filter`, or `perspective` set. Two of my seats (top center and bottom center, `data-seat='3'` and `data-seat='0'`) used `transform: translateX(-50%)` for centering; the *currently active actor* seat uses `transform: scale(1.05)` for the highlight. So three of the six seats — and a different three on every hand — anchored the menu inside the seat instead of inside the viewport.

Fix: a tiny `portal` Svelte action that appends the menu node to `document.body` on mount and removes it on destroy. The menu is now a top-level body child, no transformed ancestor in sight. Also clamped `menuX/menuY` so the menu doesn't spill past the right/bottom edges.

This is the kind of CSS gotcha that's obvious in hindsight but easy to miss — "fixed" is misleading naming. Worth the muscle memory: any time I want a popover anchored to the cursor, *portal it to the body*.

### Stale variable name after rename

After splitting `button_seat` into `button_engine_seat` / `button_ui_seat`, I missed one usage in the bot-observation call. Surfaced as a `NameError` on the first hand when the runner crashed. Trivial to fix, but the runner crashed silently in the background — only the WS log showed the connection still alive but no events flowing. Lesson restated: when refactoring a name with multiple semantic meanings, grep harder.

## Architectural note: the seat-translation abstraction

This kind of thing is why I don't regret yesterday's "small dumb bot interface" decision. Today's UI-seat / engine-seat split, sit-out, and engine resizing all happened entirely in `runner.py` and `events.py`. The bots didn't change. The frontend changed only insofar as it gained a new field per seat (`sitting_out`) and a new event type (`seats_update`). The protocol stayed simple, the layers stayed clean.

When I look at the runner now it's still the central asyncio loop yesterday, but the per-hand setup has grown a real "session-vs-hand" distinction: `self._bots` and `self._sitting_out` live across hands; `active_ui`, `e2u`, `n_active` are per-hand. That split is going to keep paying off when we add things like elimination, mid-hand chat from the LLM, or a human seat.
