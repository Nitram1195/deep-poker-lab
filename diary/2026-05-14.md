## Why the runner-reuse was almost free

The benchmark is a `GameRunner` with two things changed:

- `broadcast` is an `async def noop(_): pass`.
- A new `delay_scale: float = 1.0` constructor arg, multiplied into all five
  `await asyncio.sleep(...)` calls. Pass `0.0` and the visual pacing
  evaporates.

That's it. No fork, no parallel "headless runner" class, no
`if benchmark: skip_sleep` branches. The same code path drives both the
browser and the CLI; the table reaches the same `HandEnd` event in both modes,
the leaderboard ticks the same way, VPIP/PFR are tracked once.

This worked because the runner already had `broadcast` injected as a
constructor parameter (from way back when `ConnectionManager` was added on day
1). The benchmark just plugs in a different broadcast. Same trick FastAPI's
test client uses against `app`. The whole thing is roughly the win of an hour.

The only public-API addition was `play_hands(n)` — a tiny bounded loop next to
`_run_forever()` so benchmark can ask for exactly N and return. Later replaced
in the benchmark itself by an inline per-hand loop, because that's the level
of granularity needed for graceful Ctrl+C (see below).

## VPIP / PFR

Two counters per bot. The hand loop maintains two sets
(`vpip_engine_seats`, `pfr_engine_seats`) that get filled at each preflop
action:

```python
if street_at_action == 0:
    if action.kind == "raise_to":
        vpip_engine_seats.add(actor)
        pfr_engine_seats.add(actor)
    elif action.kind == "check_call" and legal.to_call > 0:
        vpip_engine_seats.add(actor)
```

At hand end, each bot's counters get incremented if they're in either set.
`LeaderboardEntry` grew two new fields (`vpip: int = 0`, `pfr: int = 0`,
defaulted so older clients aren't affected).

The trickier bit was deciding what VPIP *means* — the convention is "BB walks
don't count" (the BB checking their option without ever facing a raise is not
voluntary action), and that fell out naturally from `legal.to_call > 0` on
`check_call`.

## What the benchmark actually says

Sample 200-hand run (default 4 bots, no LLM):

```
Bot           Hands       Net    bb/100   VPIP    PFR
-----------------------------------------------------
GTO             200      +879   +219.75    24%    22%
Equity          200      +633   +158.25    28%    10%
TightAggro      200      +401   +100.25    10%     6%
Random          200     -1913   -478.25    57%    32%
```

A few quick observations from running it a couple of times:

- **GTO has the best VPIP/PFR profile** (`24/22` — tight-aggressive, close to a
  human TAG). Equity is loose-passive (`28/10`); it limps and calls because
  its MC equity says the price is right but it can't bluff. TightAggro lives
  up to its name. Random is the dictionary-definition maniac.
- **Margins are not stable at 200 hands**. Across runs, GTO and Equity trade
  the lead. With pokerkit's variance per hand at ~10–15bb, 200 hands of bb/100
  is mostly noise. Need ~5000+ before the ordering settles.
- **Throughput is 2 hands/s**, dominated entirely by `EquityBot`'s 120 Monte
  Carlo simulations per decision. Drop `n_simulations` to ~30 and the
  benchmark runs ~5x faster at the cost of slightly noisier equity. The other
  three bots (Random, TightAggro, GTO) are basically free.

That last point matters: if I later add a CFR-trained bot, the inference cost
will set the new ceiling. Today the "lab" runs at human-watchable pace by
*choice*, but if it had to, it could do thousands of hands per minute.

## Ctrl+C handling

Default `asyncio.run()` lets `KeyboardInterrupt` rip up out through the main
coroutine — you get a traceback and lose whatever stats the runner had
accumulated. Wrong for a benchmark where the half-completed table is exactly
what you want printed.

Replaced it with a SIGINT handler that flips a `_stop_requested` flag and
warns the user. The per-hand loop checks the flag between hands:

```python
def _handler(_sig, _frame):
    global _stop_requested
    if _stop_requested:
        raise KeyboardInterrupt   # second Ctrl-C → real exit
    _stop_requested = True
    print("\n^C — finishing current hand, then printing results...", ...)
```

Second Ctrl+C escalates to a real interrupt, in case a hand is stuck. Response
latency is bounded by one hand (~0.5s with the current bots), which is fine.

Also added a progress line every 100 hands to stderr:

```
  ... 100/500 hands  (2.0 h/s, ETA 200s)
```

Stderr deliberately, so stdout (the final table) stays clean for piping
into a file.

## A small process note

Halfway through the session I added VPIP/PFR columns to the frontend
leaderboard. User rolled those changes back across four files, kept the
backend-only version. The live UI stays minimal; the *benchmark* is the
dashboard for those stats. That ends up being a good split — the live table
is for watching individual hands play out, the benchmark is for measuring bot
quality. Trying to show both at once was overloading the same surface.

Worth remembering for future features: not every backend signal needs a
matching UI element.

## Open questions / next up

  decisive at 500 hands when they're statistically tied. Welford's online
  algorithm on per-hand pnl → SEM → 95% CI. A few hours.
- **Head-to-head matrix.** Run every pair of bots as a 2-player table, print a
  bb/100 grid. Lets you spot rock-paper-scissors (the bot that beats the field
  isn't always the one that beats the next-strongest 1-on-1).
