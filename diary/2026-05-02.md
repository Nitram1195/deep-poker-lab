## What was already in place when I started

The day's biggest piece — `HumanBot` — was authored offline between yesterday's session and today's. It's worth describing because it shaped everything else:

- **`backend/bots/human.py`**: `HumanBot.act()` is async and `await`s an `asyncio.Future` that gets resolved by a `submit(action)` call. The runner now calls `await result if inspect.isawaitable(result) else result` after `bot.act(obs)`, so sync and async bots coexist on the same loop.
- **WS becomes more bidirectional**: `{cmd: "act", seat, kind, amount}` validates the seat is human and routes the action to `HumanBot.submit()`.
- **Mid-hand resync** via a new `HandSync` event: a tab opening mid-hand previously had to wait for the next `HandStart`; now the runner stashes per-hand state (`_cur_engine`, `_cur_e2u`, `_cur_actor_ui`, `_cur_legal`, …) and ships a complete snapshot to the new viewer on connect.
- **Information hiding**: when any human is seated, opponents' `hole_cards` are zeroed in `HandStart`/`HandSync`, and street-deal `hand_labels` are filtered to the human's seat only — the server stops leaking information it shouldn't.
- **Frontend action panel**: when it's the human's turn (`is_human && isActor && legal !== null`), Fold / Call|Check / Raise-to buttons appear inside the seat with a number input clamped to `[min_raise, max_raise]`. Clicking sends the `act` command.

This was the right shape for the next changes.

## Two showdown bugs I caught only because the human seat made them visible

### Bug 1: opponent cards never revealed at showdown

Symptom: after the river, the human only saw their own cards — opponents stayed face-down even on hands that went to showdown.

Cause: the post-loop revealed dict was

```python
folded = engine.folded()
revealed = {e2u[i]: list(engine.hole_cards(i)) for i in range(n_active) if not folded[i]}
```

But pokerkit's `HAND_KILLING` automation runs at showdown and marks losing hands as inactive. By the time we read `engine.folded()`, every loser registers as folded — so `revealed` only contained the winner. The reveal had been broken since day 1; we just hadn't noticed because there was no human watching, and bots don't care about the visible card UI.

Fix: track folds via our own `actions_this_hand` history, not `engine.folded()`. And cache hole cards at hand start so a mucked loser still has cards to send to the UI:

```python
hole_by_engine_seat = {i: list(engine.hole_cards(i)) for i in range(n_active)}
...
folded_engine_seats = {h.seat for h in actions_this_hand if h.kind == "fold"}
showdown_engine_seats = [i for i in range(n_active) if i not in folded_engine_seats]
revealed_ui = {e2u[i]: hole_by_engine_seat[i] for i in showdown_engine_seats}
```

Lesson: when wrapping a third-party state machine, don't trust state observed *after* its automations have run to mean what it meant *before* them. Same kind of bug as yesterday's `RUNOUT_COUNT_SELECTION` — the engine has already moved past the point you're trying to inspect.

### Bug 2: no all-in runout shown

Symptom: when someone went all-in and was called, the UI jumped from the all-in action straight to the final state — no "ALL-IN" indicator, no flop/turn/river paced out for the rail to watch.

Cause: pokerkit's `RUNOUT_COUNT_SELECTION` (which we added on day 1 to fix the chip-conservation bug) auto-deals all remaining streets internally as soon as the all-in resolves. The runner's `StreetDeal` broadcast lives at the *top* of the action loop ("did a new street start since the last actor?"), and after the call the loop exits because `current_actor()` is `None`. So the auto-dealt streets never get broadcast — they only show up in `HandEnd.board` as a fait accompli.

Fix: after the loop exits, replay each missed street as its own `StreetDeal` with `STREET_DELAY_S` between them, plus a `Showdown` event up front so the racing hands are visible:

```python
final_board = engine.board_cards()
final_street = {0: 0, 3: 1, 4: 2, 5: 3}.get(len(final_board), 0)
if is_showdown:
    await self._broadcast(Showdown(hole_cards=revealed_ui))
    await asyncio.sleep(STREET_DELAY_S)
if is_showdown and final_street > last_street:
    for street_idx in range(last_street + 1, final_street + 1):
        # ... build partial board, recompute hand_labels, broadcast
```

Per-street hand labels are recomputed against the partial board using the cached hole cards (engine's labels would be wrong since its board is the full final one).

UI side, a pulsing red **ALL-IN** badge appears when `!folded && !sitting_out && stack === 0`, and the action label gets `(all-in)` appended.

Lesson worth keeping: bugs that are statistically harmless ("the winner's cards are still shown") become loudly visible the moment someone is actually trying to learn from the table. The human bot is also a form of QA.

## The big new thing: replay last hand

> "I want to be able to replay the last game with showed cards. so I can learn."

The shape that ended up working:

**Backend records an unfiltered event log per hand.** A new `_emit(ev_log, ev_broadcast=None)` helper appends the *unfiltered* version of every hand event (HandStart, ActorTurn, ActionEvent, StreetDeal, Showdown, HandEnd) to `_cur_replay`, then broadcasts the live (possibly information-hidden) version. For most events the two are identical; for `HandStart` and the in-loop `StreetDeal` they differ — the replay copy has every active seat's real cards and every non-folded seat's hand label. At hand-end, `_cur_replay` is promoted to `_last_replay`.

**New event type**: `HandReplay { type: "hand_replay", hand_id, events: list[dict] }`. The events are pre-`model_dump()`'d so the client can feed them straight back through its existing reducers without inventing a parallel schema.

**New WS command** `{cmd: "replay_last"}`: server responds with the `HandReplay` payload (or silently no-ops if no hand has finished yet).

**Frontend dual state.** The store now holds two `TableState`s: `liveState` (mutated by every live event as before) and an optional `replayState`. The public `state` getter returns `replayState ?? liveState`, so every component binding (`store.state.seats`, `store.state.board`, …) automatically renders the active view. All the `apply*` reducers were refactored to take a target state parameter so they can run against either.

**Replay playback** is just a loop that fires the recorded events through `applyTo(this.replayState, ev)` with delays based on the *previous* event type:

```ts
const REPLAY_DELAY_MS = {
    hand_start: 200,
    actor_turn: 700,
    action: 1200,
    street_deal: 1500,
    showdown: 1500,
    hand_end: 0,
};
```

Those numbers mirror the runner's `TURN_DELAY_S` / `ACTION_DELAY_S` / `STREET_DELAY_S`, so replay feels paced like live play. The user can hit Stop to return to live; `replayState` is dropped and the getter falls back to `liveState`, which has been quietly updating in the background the whole time.

**UI**: a "▶ Replay last hand" button in the toolbar (disabled until the first `hand_end` arrives), a magenta-glowing "REPLAY · Hand #N · all cards revealed" banner during playback, and a "■ Stop replay" button that swaps in.

The thing I'm happiest with here: the existing reducers are reused verbatim. There is no second rendering path for the replayed state. Live and replay are two state objects, one apply function, two views.

## Bugs / surprises while building replay

### The WS smoke test that wouldn't terminate

I wrote a quick Python WS client to verify `replay_last` end-to-end without the browser. It just hung. The cause was obvious in hindsight: the `HumanBot` is at seat 1 and the runner blocks on `await self._pending` until something submits an action. My test sat the human out, but `sit_out` only takes effect on the *next* hand — and the *current* hand was already mid-action and waiting forever for the human. Deadlock.

I left the test aside and verified everything in the browser. But it's a useful pattern to flag: any synthetic harness that wants to drive the runner end-to-end either needs to act for the human, or never seat one in the first place.

### `pkill -f` is too aggressive

While iterating on the WS test, I tried `pkill -f replay_test.py` to clean up between runs. Every subsequent attempt to run the test exited with code 144 (SIGTERM-ish). Eventually realized: my own command line included `replay_test`, so `pkill -f` was matching and killing the *new* process I'd just started. Anti-pattern in scripted environments where the foreground command can match its own predicate.

## Architectural note: live vs replay state, and why it's clean

The replay feature is the first time the frontend has had to render something other than live state, and the refactor turned out to be small because of yesterday's structural decisions:

- The store was already a single source of truth fed by reducers.
- The reducers were already pure functions of an event over a state object — they just hard-coded `this.state` as that object.
- Generalizing `apply` → `applyTo(target, ev)` was mechanical.

If the store had been a tangle of bespoke event handlers each touching `this.state` directly without a uniform shape, this would have been a rewrite rather than a refactor.

Backend mirrors the same idea: `_emit(ev_log, ev_broadcast=None)` is the ceiling of clever. Most call sites pass one event, replay records the same thing that goes out live; two call sites (HandStart, in-loop StreetDeal) pass two and the live version is the redacted one. No conditionals branching on "are we recording" anywhere.

## Open questions / next up

- **Replay controls.** Right now you can start and stop. Natural next features: pause/resume, slow-down or speed-up, step through one event at a time. None are hard given the current shape.
- **Multi-hand history.** User explicitly chose "just the last hand" today. Bumping this to a list of the last N hands is a 10-line change on the server (deque) and a small UI addition (pick from a list).
- **No human turn timeout.** Walk away and the table stalls forever — sit-out is the only escape hatch. A sensible auto-fold or auto-sit-out after, say, 60s would close that loop.
- **Bluffing classical bot still pending.** Now that the human is at the table, fishing for bluffs from a `PolarizedBot` would test the human's reads in a way the all-value-betting baselines can't.
- **Cross-LLM comparison still pending.** The plumbing is ready (`LLMBot` is provider-agnostic), just need a second key and a tiny config switch.
- **Hand-history persistence still pending.** The replay buffer is now structured exactly the way a JSONL log line should be — saving `_last_replay` to disk on every `HandEnd` would be the start of a real corpus.
