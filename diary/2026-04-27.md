# 2026-04-27 — Day 1: from zero to five bots playing on a live web page

## Where we started, where we ended

Started with an empty directory `/home/martin/data/deep_poker_lab` and "I want to make a poker site where bots play and I can compare them." Ended with:

- A FastAPI + SvelteKit app running locally, broadcasting a live multi-bot No-Limit Hold'em game over WebSockets
- Five bots at the table: `Random`, `AlwaysCall`, `TightAggro`, `Equity`, `groq:gpt-oss-120b`
- Hole cards face-up, hand-strength labels ("Pair of 8s", "Aces full of Ks"), spotlit current actor, live leaderboard
- Three commits, one already pushed to GitHub

The architecture choice that paid off: every bot is a single small file implementing one method (`act(observation) -> Action`). The runner, engine, event protocol, and frontend don't care which is which. That's why dropping in the LLM bot at the end was 100 lines instead of a refactor.

## What got built (in build order)

### Stack scaffolding
- Python 3.12 + `uv` for everything backend. `uv` was new to me on this machine; it pulls its own Python interpreter automatically, which sidestepped the system-Python-is-3.10-but-pokerkit-needs-3.11+ problem.
- Node had to be installed manually (no apt/sudo) — grabbed the Node 22 tarball into `~/.local/`. Then immediately had to upgrade from 22.11 to 22.22 because SvelteKit's vite plugin requires `node >= 22.12`. Lesson: always grab `latest-v22.x` not a specific patch.
- SvelteKit minimal + TypeScript + Svelte 5 (runes) for the frontend. Picked Svelte over React on the assumption of less webdev overhead; no regrets so far.

### Engine: thin pokerkit wrapper
[`backend/engine.py`](../backend/engine.py) wraps `pokerkit.NoLimitTexasHoldem` and exposes a clean per-hand API: `start`, `current_actor`, `legal_actions`, `apply`, `is_complete`, `payoffs`, `public_state`, `hand_labels_by_seat`. We don't reimplement poker rules — the rule of "pokerkit is the source of truth for legal actions" simplified everything downstream.

### Three baseline bots
- `RandomBot`: uniformly random over legal actions. Stress-tests the engine wiring and fills "every code path" coverage almost for free.
- `AlwaysCallBot`: 8 lines. The "calling station" baseline — anyone who can think should beat this.
- `TightAggroRuleBot`: hand-coded preflop range + half-pot c-bet on top-pair-or-better. Encodes "play tight, bet your made hands" wisdom.

### GameRunner + WebSocket layer
[`backend/runner.py`](../backend/runner.py) is the asyncio loop: rotate the button each hand, deal, walk the action, broadcast events, track per-bot lifetime P&L. The `broadcast(event)` callback is *injected* — the runner doesn't know FastAPI exists. That made the runner unit-testable in 20 lines.

The WS layer ([`backend/ws.py`](../backend/ws.py), [`backend/main.py`](../backend/main.py)) is a small `ConnectionManager` that broadcasts JSON to every connected viewer. Every browser tab is a spectator of the same shared table — no per-client state.

### Frontend: SvelteKit + reactive store
[`frontend/src/lib/store.svelte.ts`](../frontend/src/lib/store.svelte.ts) is a class with a `$state` field that ingests events and projects them to a `TableState`. Components are pure projections of the state — no callbacks, no event handlers. The whole UI is driven by the WS event stream.

`PokerTable` lays out 6 seats around an oval green felt. `Seat` shows name, stack, bet, hole cards, last action, hand-strength label. `Card` renders any card by `RANK + SUIT` string. `Leaderboard` table at the bottom.

### EquityBot — first principled bot
[`backend/bots/equity.py`](../backend/bots/equity.py): for each decision, Monte-Carlo the hand to showdown 120 times against random opponents, then act on pot odds. Fold if equity < pot_odds + 3% margin; raise pot if equity > 75%; raise half-pot if equity > 60%; preflop open-raise if equity > 50%; otherwise call.

Uses `pokerkit.StandardHighHand` for hand evaluation (~1750 evals/sec on this machine). 120 sims × ~6 evals/sim = ~280 ms per decision. Hidden inside the existing 800 ms visual delay.

In a 277-hand self-play test:

| Bot | bb/hand |
|---|---|
| **Equity** | **+8.82** |
| TightAggro | +2.85 |
| Random | -1.44 |
| AlwaysCall | -10.23 |

The order is what theory predicts (Equity > TightAggro > Random > AlwaysCall), and AlwaysCall getting crushed by Equity is exactly the textbook outcome of value-betting a calling station.

### Hand-strength labels
Backend computes per-seat labels at every street change and at hand-end. Each label has both descriptive text ("Pair of 8s") and a raw category ("One pair"). The frontend ties the category to a color tier (gray/blue/green/yellow/orange/red as the hand gets stronger). Glance at the table during a hand and the brightest-color chip is who's leading.

The wheel (A-2-3-4-5) and Royal flush are special-cased so the text reads naturally.

### Pacing + visual highlight overhaul
The first version flashed events too fast and the actor highlight only showed for milliseconds. Two fixes:

- Backend: introduced `TURN_DELAY_S` so the actor's seat is highlighted for a beat *before* the bot's `act()` is called. Also slowed the post-action / post-street / between-hand delays.
- Frontend: don't clear `current_actor` when an action lands — let the *next* `actor_turn` move the spotlight. Combined with a pulsing yellow outline, a `▶` arrow next to the bot's name, and a 4% scale-up, the active seat is now genuinely impossible to miss.

### LLMBot — generic OpenAI-compatible bot
[`backend/bots/llm.py`](../backend/bots/llm.py) is provider-agnostic: takes `base_url`, `api_key`, `model`. Same code works against Groq, OpenAI, OpenRouter, Ollama, anything OpenAI-compatible. Sends a structured prompt (system: rules + personality, user: game state), expects JSON action back, parses leniently (clamps amounts to legal range, accepts `"call"`/`"check"`/`"raise"` as variants), falls back to `check_call`/`fold` on parse or API errors.

Wired into `main.py` conditionally — only joins the table when `GROQ_API_KEY` is in the environment. Default model: `openai/gpt-oss-120b` (overridable via `GROQ_MODEL`).

After plugging in a Groq key, the LLM bot played its first hands successfully. Mostly check/calls in the small sample observed — too early to judge style.

## Bugs / surprises

### The chip-conservation bug (most interesting)

Built the runner, ran 100 hands, total chip P&L was -400 instead of 0. Engine alone (not via runner) was conserving fine, so it had to be runner-side. Added an assertion that every `HandEnd`'s payoffs sum to zero — instantly found hands like `{0: -200, 1: -200, 2: 0}`: two players each lost their full stack, but seat 2 didn't gain anything. Chips vanishing into the void.

Root cause: pokerkit has an automation called `RUNOUT_COUNT_SELECTION` that handles the "you're allowed to run the board multiple times after an all-in" decision. I'd enabled every other automation but missed this one. Without it, all-in hands stalled with `actor_index == None` *while* `status == True`, my runner broke out of the loop early, and read `payoffs` before the chips had been pulled.

Fix: one line — add the missing automation. After the fix, 500 random hands all conserved.

Lesson: when wrapping a third-party state machine, don't assume "the API returns the obvious shape" — assert chip conservation as a property test from day one. It would have caught this in seconds instead of after building most of the runner.

### Frontend SSR reading `location`

First page load returned HTTP 500 with "Internal Error". The code referenced `location.protocol` at module-init time, which fails on the server side during SSR (Svelte renders on the server first). Moved the lookup into `onMount`. Easy to forget when coming from non-SSR frameworks.

### EquityBot performance

`pokerkit.StandardHighHand` does ~1750 evaluations/sec. With 120 sims × ~6 evals/sim, that's ~280 ms per decision. I worried this would block the asyncio loop noticeably, but it slots inside the existing `TURN_DELAY_S` and is invisible to the user. If we ever want to scale up sim count or have multiple equity bots, we'd run it in a thread executor.

### Surprise: AlwaysCall < Random

When EquityBot was added, the leaderboard ordered them: Equity > TightAggro > Random > AlwaysCall. Initially I expected Random to lose to AlwaysCall (the "do nothing" bot should be safer than randomly shoving). But `EquityBot` value-bets AlwaysCall relentlessly when ahead — and AlwaysCall *never folds*, so it pays off every value bet — while Random's randomness occasionally wins by sheer card luck. This is the lab earning its keep: the leaderboard surfaced an interaction effect I didn't predict from the bot designs alone.

## Late-day iteration: making the LLM bot actually play

After watching for a while, the LLM was very passive — almost all check/calls. Three issues, all LLM-side:

1. **No action history in the prompt.** The model could see the static board state but not who raised, who 3-bet, or what street the action was on. So it had no way to react to aggression or recognize a c-bet pattern. This was the single biggest gap.
2. **Too gentle a personality.** "Strong but unpredictable" is a bland prompt; the model defaulted to a tight calling style.
3. **The default model was wrong.** `openai/gpt-oss-120b` is a *reasoning* model — it spends hidden chain-of-thought tokens before producing the JSON answer. With my `max_tokens=150`, ~30% of calls returned empty content (Groq then fails JSON-mode validation server-side with `failed_generation: ''`). Bumping to 1500 helped but didn't fully fix it.

Fixes:

- **Action history threaded through.** Added `HistoryEntry` to `Observation`, the runner now appends one per action and passes the list through `build_observation`. The LLM prompt formats it grouped by street: `preflop: seat 1 (TightAggro) raises to 6, seat 2 (groq) checks/calls, ...`. Pure additive change — other bots don't have to read it.
- **Sharper personality.** "Sharp, exploitative — value-bet calling stations relentlessly, semi-bluff with draws, bluff occasionally on scary boards or after opponents show weakness, vary your sizing."
- **`temperature 0.7 → 0.9`** for more variety. **`max_tokens 150 → 1500`**, **`timeout 10 → 20`**.
- **Default model switched** to `llama-3.3-70b-versatile` (non-reasoning instruct model, reliable JSON output). `openai/gpt-oss-120b` still works as `GROQ_MODEL=openai/gpt-oss-120b` for users who want to try the reasoning model.

Result, watching 12 LLM actions: **8 raises, 3 folds, 1 check_call** — and varied sizing (raises of 6, 8, 12, 15, 24, 40). Night-and-day from the previous "everything is check_call" sample. Whether it actually wins money long-term is a different question — needs sample size.

The lesson on reasoning models is generic and worth remembering: when a provider exposes a reasoning model behind an OpenAI-compatible API, JSON mode + tight `max_tokens` is a footgun. Either go non-reasoning or budget enough tokens for both the chain-of-thought and the answer.

## Open questions / next up

- **LLM bot style.** Need more hands of `groq:gpt-oss-120b` to characterize how it plays. Early sample suggests passive (lots of check/calls). Knobs to try: bump `temperature`, beef up the personality prompt to encourage exploitative play and bluffing.
- **Cross-LLM comparison.** The whole point of the lab. Add 2–3 more LLMs (different model families) and watch them play each other and the classical bots.
- **Bluffing classical bot.** Even with the LLM at the table, the classical lineup is all "value-only." A `PolarizedBot` (bet pot with strong, bluff with very weak, check medium) would close that gap and create an interesting interaction with whatever bluffs the LLM throws.
- **Action history in the prompt.** The current LLM prompt doesn't include the action history of the current hand, only the static state. That's a real gap — the LLM can't tell if seat 3 raised three streets in a row vs called everything down. Adding history is straightforward and probably the single biggest improvement available.
- **Persistence.** Hand histories are currently nowhere. To do real bot evaluation across thousands of hands, we'll want to log hands to a file or DB.
- **Rotate the Groq key** when done — the one used today is in this conversation transcript.
